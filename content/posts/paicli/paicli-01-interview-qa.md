# MINI-CLI 项目面试 Q&A

> 来源：简历第二个项目 `MINI-CLI` + 当前 PaiCLI 仓库实现。定位是后端实习生面试中的项目深挖复习稿，答案按“能直接说出口”的口径组织。

## 项目一句话介绍

MINI-CLI 是一个仿 Claude Code 的本地 Agent CLI，技术栈是 Java 17、Maven、OkHttp、SQLite 和 Jieba。项目核心不是简单聊天，而是让大模型可以通过 ReAct、Plan-and-Execute、Multi-Agent、Memory、RAG 和并行工具执行去理解代码库、调用工具、拆解复杂任务并完成本地开发工作流。

---

## Q1. 你这个 MINI-CLI 项目整体解决了什么问题？为什么不只是接一个大模型 API？

**答案：**

这个项目解决的是“让大模型真正参与本地开发任务”的问题，而不是只做普通聊天。普通大模型 API 只能根据输入文本回答，但开发任务需要读文件、改文件、执行命令、理解代码库、跨多轮保存上下文，还要能把复杂任务拆成步骤。

所以我把系统设计成一个本地 Agent CLI：用户输入任务后，Agent 可以基于工具调用读取项目文件、搜索代码、执行命令，并把观察结果回灌给模型继续决策。对于简单任务走 ReAct 循环；对于复杂任务走 Plan-and-Execute，先规划 DAG 再按依赖执行；对于需要角色分工的任务走 Multi-Agent，由 Planner、Worker、Reviewer 协作。

项目里还做了 Memory、RAG、并行工具执行和策略层安全控制。这样它更像一个可落地的开发助手，而不是“API 包一层壳”。

---

## Q2. ReAct 在你的项目里是怎么实现的？完整链路是什么？

**答案：**

ReAct 的核心链路是“思考、行动、观察、再思考”。在我的实现里，入口是 `Agent.run(userInput)`。收到用户输入后，会先写入短期记忆，然后检索相关长期记忆，更新 system prompt，再把用户输入追加到 `conversationHistory`。

之后进入循环：每一轮把当前对话历史和工具定义传给 LLM。如果 LLM 返回 tool calls，就把 assistant 消息和工具调用记录进历史，然后调用 `ToolRegistry.executeTools()` 执行工具。工具结果会作为 tool message 回灌到 `conversationHistory`，同时进入短期记忆。然后继续下一轮，让模型基于观察结果决定下一步。

如果某一轮 LLM 不再返回工具调用，就说明任务完成，Agent 把最终回答加入历史和短期记忆，然后结束。本质上退出条件由模型决定，工程上再用 token 预算、重复工具调用检测、最大迭代次数和取消标记做兜底。

---

## Q3. ReAct 模式为什么可能失控？你做了哪些兜底？

**答案：**

ReAct 的风险主要有三个：第一，模型可能不断调用同一个工具进入死循环；第二，对话历史和工具结果越来越长，超过上下文窗口；第三，工具执行可能失败、超时或者被用户取消。

我做了几层兜底。第一是 `AgentBudget`，它会记录迭代轮数、token 使用和工具调用模式，项目里有硬最大迭代轮次，连续重复工具调用也会触发退出。第二是上下文压缩，短期记忆和真正发给模型的 conversation history 是两道不同压缩，避免只压 memory 但请求历史仍然爆掉。第三是工具层超时和取消，`execute_command` 有命令级超时，批量工具调用也有批次超时，运行中可以通过取消上下文中断后续执行。

面试里我会强调：ReAct 不是只写一个 while 循环，真正难点在退出条件、上下文控制和工具失败后的稳定性。

## Q4. Plan-and-Execute 和 ReAct 有什么区别？你为什么还要做 Plan + DAG？

**答案：**

ReAct 更适合短任务或探索性任务，它是一边执行一边想下一步。缺点是对复杂任务的全局依赖关系把握不稳定，比如“先改模块 A，再补测试，再跑回归，再修失败”这种任务，如果没有整体计划，容易漏步骤或顺序错。

Plan-and-Execute 是先规划再执行。我的项目里由 `Planner` 生成 `ExecutionPlan`，计划里包含多个任务和依赖关系。执行时先经过用户审阅，用户可以确认、取消或补充反馈。真正执行阶段会按 DAG 依赖推进，依赖满足的任务才会执行，互相独立的任务可以并行。

所以两者的定位不同：ReAct 灵活，适合即时推理；Plan + DAG 更稳，适合多步骤、强依赖、需要用户确认的复杂开发任务。

---

## Q5. 你说 Plan 支持 DAG，具体怎么保证执行顺序？

**答案：**

DAG 的核心是把任务看成节点，把依赖关系看成边。执行时不是简单按列表顺序跑，而是每一轮找出“依赖都已经完成”的任务作为可执行集合。

比如任务 B 依赖任务 A，那么 A 没完成前 B 不能执行；任务 C 和 D 都只依赖 A，那么 A 完成后 C、D 可以进入同一个批次并行执行。这样既保证依赖正确，又能利用并行提升效率。

我的回答重点会放在两个点：第一，计划数据结构要显式记录 task id 和 dependencies；第二，执行器要维护任务状态，比如 pending、running、completed、failed。只要前置失败，后续依赖任务就不能盲目执行，要给出跳过或失败原因。

---

## Q6. Multi-Agent 在你的项目里怎么设计？为什么要分 Planner / Worker / Reviewer？

**答案：**

Multi-Agent 是为了把“规划、执行、审查”三个职责拆开，减少单个 Agent 同时做所有事情带来的不稳定。我的实现是主从架构，`AgentOrchestrator` 作为编排器，下面有 Planner、两个 Worker 和 Reviewer。

流程是：用户提交任务后，Planner 先输出结构化计划；Orchestrator 解析计划并按依赖找出可执行步骤；Worker 负责具体执行步骤；Reviewer 对 Worker 的结果做质量检查。如果 Reviewer 不通过，会带反馈让 Worker 重试，单步最多重试 2 次。

这样做的好处是职责清晰。Planner 关注拆解，Worker 关注落地，Reviewer 关注质量。对于复杂任务，比一个 Agent 自己边做边检查更容易发现遗漏。

---

## Q7. Multi-Agent 并行执行时，多个 Worker 同时输出会不会把终端打乱？你怎么处理？

**答案：**

会，这是 CLI 类项目很实际的问题。如果多个线程同时写同一个终端输出流，用户看到的日志会交错，甚至影响 TUI 布局。

我的处理方式是：同一依赖批次里多个步骤可以并行执行，但每个步骤使用独立的缓冲输出流。等这一批任务完成后，再按 step id 或原始顺序把缓冲内容 flush 到主输出。这样内部执行是并行的，外部展示仍然有序。

单步批次则不需要缓冲，可以直接流式输出，保留实时“打字”的体验。这个取舍兼顾了并发效率和终端可读性。

---

## Q8. 你实现的并行工具调用具体怎么做？为什么结果还要按原始顺序回灌？

**答案：**

工具调用统一走 `ToolRegistry.executeTools()`。如果同一轮 LLM 返回多个 tool calls，工具层会创建固定大小线程池，最大并发数是 4，把每个工具调用包装成 `Callable` 并行执行。

结果必须按原始 tool call 顺序返回，因为 LLM 的消息协议里，assistant 发出的 tool call 和后续 tool result 是一一对应的。如果乱序回灌，模型可能把 A 工具的结果理解成 B 工具的结果，导致下一轮推理错误。

所以实现上并行只是执行层优化，语义上仍保持顺序一致。面试里可以总结成一句话：并发提升吞吐，但协议顺序不能破坏。

---

## Q9. 并行工具调用会带来什么线程安全问题？

**答案：**

主要有四类风险。第一，共享状态并发修改，比如工具注册表、记忆系统、审计日志等。第二，多个工具同时写同一个文件，可能覆盖彼此结果。第三，多个工具同时输出到终端，造成输出交错。第四，超时取消时，部分工具完成、部分工具失败，需要保证结果可解释。

我的项目里对工具注册结构使用并发容器，比如 `ConcurrentHashMap`；工具结果按顺序收集；终端输出在 Multi-Agent 并行路径中用缓冲避免交错；批量工具有超时控制。对于文件写冲突，目前更依赖模型规划和上层任务拆解避免同一文件并行写，这是后续可以加强的点，比如加文件级锁或在 Planner 阶段识别冲突。

---

## Q10. Memory 架构为什么要区分短期记忆、长期记忆和上下文压缩？

**答案：**

因为这三者解决的问题不一样。短期记忆保存当前会话里的用户消息、助手回答和工具结果，目标是让 Agent 在本轮或近期任务里不断续上上下文。长期记忆保存跨会话稳定事实，比如用户偏好、项目固定约定，它不应该自动乱存，而是通过 `/save` 或用户明确要求保存。上下文压缩解决的是大模型窗口有限的问题，把过长历史摘要化。

我项目里 `MemoryManager` 是门面，内部有 `ConversationMemory`、`LongTermMemory`、`ContextCompressor` 和 `MemoryRetriever`。工具结果进入短期记忆时会截断，避免大日志快速撑爆预算。长期记忆只保存明确事实，防止把一次性任务当成永久偏好。

这个设计的重点是：记忆不是越多越好，而是要分层、可控、可检索。

---

## Q11. 你提到“两道压缩”，具体是哪两道？为什么不能混在一起？

**答案：**

第一道是 shortTermMemory 压缩，它发生在 MemoryManager 内部，主要控制短期记忆对象本身的 token 占用。第二道是 conversationHistory 压缩，它发生在 LLM 调用前，控制真正要发给模型的消息历史。

这两道不能混淆，因为短期记忆压缩了，不代表 `conversationHistory` 变短。LLM 实际收到的是系统提示词、用户消息、assistant 消息、tool message 等组成的历史。如果只压 memory，不压真实请求历史，窗口还是可能爆掉。

所以我把它们分开：Memory 压缩用于长期运行时的记忆管理；History 压缩用于保证下一轮 LLM 请求能放进上下文窗口。

## Q12. 你项目中的 RAG 是怎么做代码库理解的？

**答案：**

RAG 流程分为索引和检索两部分。索引阶段由 `CodeIndex` 遍历项目文件，跳过 `.git`、`target`、`node_modules` 等无关目录，只处理 Java、Python、JS、Markdown、XML、YAML 等文本代码文件。

每个文件会经过 `CodeChunker` 切成代码块，然后用 `EmbeddingClient` 生成向量，再持久化到 SQLite 的 `VectorStore`。Java 文件还会通过 `CodeAnalyzer` 分析类、方法和调用关系，写入关系表。

检索阶段，`search_code` 会把用户的自然语言 query 转成向量，在 SQLite 中取出当前项目的候选向量，内存计算余弦相似度，按 TopK 返回相关代码块。这样 Agent 不需要一次性读完整仓库，也能先定位相关文件和方法。

---

## Q13. 为什么你的向量库用 SQLite，而不是直接用 Elasticsearch、FAISS 或 pgvector？

**答案：**

这是项目定位决定的。MINI-CLI 是本地 Agent CLI，目标是轻量、易启动、少依赖。SQLite 单文件持久化非常适合本地工具，不需要用户额外启动服务。

当前代码库规模通常是几百到几千个 chunk，向量存在 SQLite 里，检索时加载候选并在内存计算余弦相似度，性能足够。FAISS、pgvector 或 Elasticsearch 更适合大规模向量检索，但会增加部署复杂度。

我会把它解释成工程权衡：本地 CLI 的第一优先级是可用和低门槛，等代码库规模、检索量或多用户场景上来，再替换成专业向量数据库。

---

## Q14. RAG 检索出来的代码片段如何避免误导模型？

**答案：**

RAG 最大的问题是召回不一定准，模型还可能过度相信检索片段。所以我会从三个层面控制。

第一，检索结果要带元信息，包括文件路径、chunk 类型、名称和相似度，让模型知道来源。第二，TopK 不宜过大，默认给少量高相关片段，长上下文模式下再适当提高。第三，Agent 不应该只依赖 RAG 结果做最终判断，关键修改前还要用 `read_file` 读取真实文件确认上下文。

如果面试官追问“怎么优化”，我会说可以做混合检索：关键词匹配精确定位类名/方法名，向量检索召回语义相关，再结合调用关系图谱扩展上下游代码。

---

## Q15. 工具调用系统怎么设计？一个工具从注册到执行的流程是什么？

**答案：**

工具统一由 `ToolRegistry` 管理。每个工具包含名称、描述、参数 schema 和 executor。启动时会注册内置工具，比如 `read_file`、`write_file`、`list_dir`、`execute_command`、`search_code`、`web_search` 等。MCP 工具也会被动态注册成统一格式。

调用流程是：LLM 根据工具定义生成 tool call，Agent 把 tool call 转成 `ToolInvocation`，交给 `ToolRegistry` 执行。执行前先解析 JSON 参数，然后进入对应工具的 executor。文件类工具会先通过 `PathGuard` 做路径校验，命令类工具会通过 `CommandGuard` 做快速拒绝，高风险工具还会进入 HITL 和审计链路。

执行结果会包装成 tool result 回灌给 LLM。这样模型只看到统一工具接口，不需要关心工具背后是本地函数、Shell 命令还是 MCP server。

---

## Q16. 本地 Agent CLI 最大的安全风险是什么？你做了哪些限制？

**答案：**

最大风险是模型可以调用本地工具，一旦没有边界，可能读写用户不希望操作的文件，执行危险命令，或者把敏感信息带出去。

我做了几层限制。第一，文件工具通过 `PathGuard` 限定在项目根目录内，绝对路径逃逸、`..` 穿越和符号链接逃逸都要拒绝。第二，`write_file` 有 5MB 单次写入上限，避免误写超大内容。第三，命令工具通过 `CommandGuard` 拦截明显危险命令，比如全盘删除、格式化磁盘、fork bomb、`curl | sh` 这类模式。第四，高风险工具会走 HITL 审批和 AuditLog 审计。

我也会主动说明：这不是完整沙箱，不能等价于容器或 VM 隔离。它是本地 CLI 的策略层防护，核心是降低误操作风险。

---

## Q17. HITL 和策略层有什么区别？为什么说用户也不能批准策略拒绝？

**答案：**

HITL 是人类审批层，解决的是“这个操作是否需要用户确认”。策略层是硬规则，解决的是“这个操作是否永远不应该执行”。

比如写项目内某个文件，这可能是中风险操作，可以让用户批准。但如果模型试图写到项目根目录外，或者执行明显破坏性的命令，这属于策略拒绝，用户不能通过 HITL 放行。否则策略层就失去意义了。

项目里的拦截顺序是 HITL、ToolRegistry、PathGuard/CommandGuard 等策略校验共同构成防线。面试时可以这样总结：HITL 管授权，Policy 管边界；授权不能突破边界。

---

## Q18. Agent 项目里 system prompt 为什么要做分层？有什么好处？

**答案：**

如果所有规则都硬编码在 Java 字符串里，后期会很难维护。不同模式，比如 ReAct、Plan、Multi-Agent，不同角色，比如 Planner、Worker、Reviewer，需要的提示词也不完全一样。

我的项目里把 prompt 做成分层组装：base、personality、mode、approval、project_context、skills、context_mgmt、handoff 等模块按顺序组合。这样通用规则放 base，模式相关规则放 mode，项目上下文和 skill 索引动态注入。

好处是可维护、可覆盖、可测试。用户级和项目级还可以覆盖 prompt 文件，不需要重新编译代码。对于 Agent 产品来说，prompt 本身就是运行时配置的一部分，不应该散落在业务代码里。

---

## Q19. Agent 的 Function Calling / Tool Calling 和普通函数调用有什么本质区别？

**答案：**

普通函数调用是程序决定什么时候调、传什么参数，调用路径是确定的。Agent 的 Tool Calling 是模型先根据上下文选择工具和参数，程序再负责校验、执行和回灌结果。

这意味着 Tool Calling 的难点不在“调用函数”本身，而在协议和控制。工具描述要让模型理解；参数 schema 要足够明确；执行前要做安全校验；执行后要把结果按正确 tool call id 回灌；失败、超时、取消也要用模型能理解的文本表达。

所以我会说，Agent 工具系统本质是“LLM 决策 + 程序执行 + 受控反馈”的闭环，不是把方法暴露出去这么简单。

---

## Q20. 如果面试官问你这个项目最大的技术难点和不足，你怎么回答？

**答案：**

我会说最大的难点是稳定性，而不是某个单点功能。Agent CLI 涉及 LLM 不确定输出、本地工具副作用、上下文膨胀、并发执行、终端渲染和安全策略。任何一层处理不好，用户体验都会很差。

我在项目里重点解决了几件事：ReAct 循环的退出兜底；Plan + DAG 的依赖执行；Multi-Agent 的角色分工和审查重试；工具并行执行但结果有序回灌；Memory 的短期、长期和历史压缩分层；RAG 的代码切片、向量存储和语义召回；以及 PathGuard、CommandGuard、HITL、AuditLog 这些安全控制。

不足我也会坦诚说：当前 SQLite 向量检索适合中小项目，大项目需要更专业的向量索引；并行写文件还可以加文件级冲突检测；策略层不是容器级沙箱；Agent 评测体系还可以补，比如用固定任务集评估成功率、工具调用次数和回归稳定性。

---

## 快速背诵版

1. 项目定位：本地 Agent CLI，不是聊天壳，而是能调用工具完成开发任务。
2. 三条执行路径：ReAct 灵活探索，Plan+DAG 稳定处理复杂依赖，Multi-Agent 做规划、执行、审查分工。
3. 工具系统：统一注册、统一 schema、统一执行，结果按 tool call 顺序回灌。
4. 并行优化：执行并行，语义有序；终端输出用缓冲避免交错。
5. Memory：短期记忆、长期记忆、conversation history 压缩分开讲。
6. RAG：代码切片、Embedding、SQLite 向量存储、余弦相似度 TopK 召回。
7. 安全：HITL 是审批，Policy 是硬边界；本地 CLI 不是完整沙箱。
8. 不足：大规模检索、并发写冲突、容器级隔离、Agent 评测体系还可继续增强。
