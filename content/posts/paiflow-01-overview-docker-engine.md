## 实现了什么

我们实现了一套基于 DAG 的执行引擎，支持条件分支、并行执行、循环节点。每个节点执行完会把输出写到变量池中，下游节点通过变量引用的方式可以拿到上游的节点数据。执行状态会持久化到数据库，如果中间某个节点失败了，支持从断点重试，不用整个流程重跑。变量池基于concurrentHashMap实现

部署这块我们做了 Docker Compose 一键启动，十几个服务的依赖关系、环境变量、网络配置全部封装好了，首次部署稍微花点时间，但所有依赖下载完成也差不多 30 分钟左右

## 单独做一个引擎的原因

工作流引擎是整个系统的"大脑"，它负责三件事：**解析流程定义、按顺序执行节点、管理执行状态。**

之所以要单独做一个引擎，而不是把执行逻辑写在业务中台里。最直接的原因是关注点分离。业务中台关心的是"用户要什么"，比如谁有权限跑这个流程、流程定义怎么存、历史记录怎么查。而引擎关心的是"怎么把流程跑起来"，比如节点之间的依赖怎么解析、并行分支怎么调度、某个节点超时了怎么处理。

- **职责分离**：Console Hub 负责用户管理、权限这些，工作流引擎专注于流程调度，各司其职

- **可扩展性**：工作流执行是 CPU 密集型的，单独部署可以独立扩容

- **技术栈灵活**：我们同时维护了 Java 和 Python 两个版本，可以根据场景切换

- **复用性**：工作流引擎的能力可以给其他业务系统复用，不只是 PaiFlow 一个产品

## 你的 WorkflowEngine 是怎么解析 DSL（JSON）并构建执行链路的？

**第一步：反序列化**，前端传过来的是 JSON，我用 FastJSON 直接反序列化成 WorkflowDSL 对象，里面包含 nodes 列表和 edges 列表。

**第二步：构建节点索引**，遍历 nodes 列表，建一个 Map<String, Node>，key 是节点 ID，方便后面快速查找。同时找出 Start 节点作为入口。

**第三步：根据边构建节点关系**，遍历 edges 列表，对于每条边：

- 把 target 节点加到 source 节点的 nextNodes 列表

- 把 source 节点加到 target 节点的 preNodes 列表

- 如果是异常分支的边，加到 failNodes 列表

工作流引擎采用了深度优先搜索 (DFS) 结合递归回溯检查的策略。当引擎尝试执行节点 C 时，它不会立即执行，而是先检查 C 的所有前置节点（A 和 B）的状态。

## 错误处理策略

我们定义了三种错误处理策略，通过 ErrorStrategyEnum 枚举表示：

ERR_INTERUPT（中断）：节点失败后，整个工作流停止，向上抛异常。适合关键节点，比如付款失败就不能继续发货

ERR_CODE（错误码）：节点失败后，用配置好的默认值作为输出，继续执行后续节点。适合非关键节点，比如日志记录失败不影响主流程

ERR_CONDITION（错误分支）：节点失败后，不走正常的 n nextNodes，改走 failNodes 里的异常处理分支。适合需要特殊处理的场景，比如调用外部 API 失败就走降级逻辑

**如何判断超时**

- 把节点执行逻辑提交到一个单独的线程

- 主线程用 Future.get(timeout) 等待

- 如果超时了，Future.get() 抛出 TimeoutException

- 捕获异常后，返回超时错误，走错误处理策略

## 设计模式的运用

**1. 策略模式（NodeExecutor）**，不同类型节点有不同的执行逻辑，通过接口多态实现

**2. 模板方法模式（AbstractNodeExecutor）**，父类定义执行骨架，子类实现具体逻辑

**4. 观察者模式（回调机制）**，节点执行状态变化通过回调通知

___

## 策略模式和模版方法模式的区别

### 追问 1：和策略模式的区别？

策略模式是"整体替换"——不同策略是完全独立的实现。模板方法是"部分替换"——骨架固定，只替换其中某些步骤。

在 NodeExecutor 的设计里，两者结合使用：**模板方法定义单个节点的执行流程，策略模式用于切换不同类型的执行器**。工作流引擎根据节点类型选择策略（哪个执行器），执行器内部用模板方法控制流程。

## 为什么是SSE

我们选择 SSE 的原因有2，第一个是**场景匹配**：大模型输出是服务端单向推给前端，不需要双向通信；第二个是**简单**：不需要额外的协议，nginx 也好配置。

**第一，连接开销小**。长轮询每次收到数据后要重新建连接，TCP 握手、HTTP 头解析都要重来一遍。SSE 一个连接用到底，省掉了大量重复开销。对于 LLM 流式输出这种一秒要推几十个 token 的场景，差距非常明显。

**第二，实时性更好**。长轮询在收到数据和发起新请求之间有时间差，可能错过这期间的数据。SSE 是持续推送，数据来了立刻就能发出去，不存在空窗期。

**第三，服务端更简单**。长轮询要维护"请求-响应"的对应关系，请求超时了要处理、客户端没及时发新请求要兜底。SSE 就是往流里写数据，逻辑清晰得多。

**第四，浏览器原生支持**。SSE 有标准的 EventSource API，自带断线重连、事件ID、消息类型这些功能，不用自己实现。
