# AI 工作流项目面试问答

## Q1：你先整体介绍一下这个 AI 工作流编排平台，你负责的部分具体是什么？

**答案：**

这个项目是一个 AI 工作流编排平台，核心目标是让用户通过可视化方式，把大模型节点、插件节点和流程控制节点编排成一条完整工作流，然后由后端引擎去解析和执行。

我在这个项目里主要负责三块。第一块是工作流引擎侧的 DAG 解析和执行链路构建，也就是把前端传来的 DSL 转成后端可执行的节点关系，并按顺序执行。第二块是基于 SSE 的实时状态同步，我把工作流开始、节点开始、处理中、结束这些状态通过回调机制实时推送给前端。第三块是节点执行器这一层的抽象设计，用工厂模式和策略模式把不同类型节点的执行逻辑解耦，方便后续扩展 LLM 节点、插件节点和其他控制类节点。

我理解这个项目的难点不只是把节点跑起来，而是既要让执行链路清晰，又要让状态同步及时，还要给后续节点扩展留出空间。

## Q2：你简历里写了“工厂模式 + 策略模式结合，规范多种节点开发流程”，请你具体讲讲这个设计。

**答案：**

这个设计主要是为了解决不同节点类型执行逻辑差异很大，如果全写在一个大 `if-else` 里，后期会非常难维护的问题。

在我们这个项目里，不同节点比如 `StartNode`、`LLMNode`、`PluginNode`，它们的执行逻辑完全不同，所以我把“每种节点怎么执行”抽象成不同的 `NodeExecutor` 实现类。比如 LLM 节点负责 prompt 渲染、历史消息拼接、模型调用和流式输出；插件节点负责调用外部工具服务；开始节点主要负责初始化输入。

策略模式体现在“每种节点有自己的执行策略”。工厂模式更准确说是注册表分发机制，体现在引擎启动时会把所有 `NodeExecutor` 按 `NodeTypeEnum` 注册进一个 `Map` 里，执行时根据节点类型直接找到对应执行器。

这样做有两个好处。第一，新增节点类型时不用改引擎主流程，基本只要新增执行器并注册即可。第二，引擎层只关心“调度哪个执行器”，不关心“执行器内部怎么做”，职责会更清晰。

## Q3：你说实现了 DAG 工作流解析引擎，那一个工作流从 DSL 到真正执行，完整链路是怎样的？

**答案：**

完整链路我一般分成四步来讲。

第一步是前置校验。后端收到工作流 DSL 后，会先校验节点和边是不是完整，比如有没有开始节点、结束节点，每个节点类型有没有对应的执行器，以及整个图里有没有环。

第二步是构建执行链路。我们会遍历所有节点先建立 `nodeMap`，然后根据边去建立节点之间的前驱、后继、失败分支关系。这样每个节点都会知道它的 `preNodes`、`nextNodes` 和 `failNodes`。

第三步是初始化执行上下文。找到开始节点后，把用户输入写入 `VariablePool`，同时初始化工作流上下文和回调对象，比如 chatId、workflowId、SSE callback 这些。

第四步才是真正执行。引擎从开始节点进入递归执行，先确保前置节点都执行完成，再调用当前节点对应的执行器。执行器返回状态后，引擎根据状态决定走正常分支还是失败分支。比如成功就继续跑 `nextNodes`，失败但有失败分支就跑 `failNodes`，如果是中断类错误就直接结束整个流程。

所以本质上，DSL 到执行不是简单“顺序遍历”，而是“图结构解析 + 状态驱动调度”。

## Q4：你简历写“基于 Kahn 算法的拓扑排序确定节点执行顺序，结合 DFS 检测循环依赖”，但如果我看你们现有 Java 实现，环检测更像是基于入度的方式，你怎么解释？

**答案：**

这个地方我会比较诚实地回答。

从设计思路上，我们讨论过两类方案。一类是基于 Kahn 算法做拓扑排序和环检测，因为它本身就基于入度统计，比较适合 DAG 这种场景；另一类是用 DFS 做环检测，它更适合从访问路径角度判断回边。

在当前 Java 版本的实现里，环检测落地主要是基于入度的方式，也就是 Kahn 算法那条思路：先统计每个节点入度，然后不断弹出入度为 0 的节点，最后看处理节点数是否等于总节点数，不相等就说明有环。

所以如果严格按代码来说，当前实现核心是 Kahn 思路，不是 DFS 主导。DFS 更多是我在设计和学习阶段考虑过的备选方案，它在做图遍历分析时也很常见，但在这个版本里没有作为主实现保留下来。

如果面试里我讲这个点，我会明确区分“设计时考虑过什么”和“当前代码实际用了什么”，避免说得太泛。

## Q5：如果一个节点执行失败，但工作流配置里给它配了失败分支，这个流程在你们引擎里是怎么继续跑下去的？

**答案：**

我们这里不是简单地“失败就结束”，而是支持节点级失败分支。

每个节点除了正常后继 `nextNodes` 之外，还可以维护失败后继 `failNodes`。当节点执行器返回结果时，引擎会看它的 `NodeExecStatusEnum`。如果是 `SUCCESS`，那就把节点状态设成成功，继续执行正常分支；如果是 `ERR_FAIL_CONDITION`，就说明这个节点失败了，但流程还要按配置进入异常处理链路，这时节点会被标记成 `ERROR`，然后执行 `failNodes`。

同时，正常分支上那些本来可能会继续执行的节点，如果这次不该走到它们，就会先标记成 `MARK` 或后续变成 `SKIP`，避免被误执行。这个状态设计主要是为了处理复杂分支场景，尤其是多前驱节点时，不能单纯靠一个布尔值判断节点该不该跑。

所以核心思路是：节点失败不是直接终止，而是根据预先定义好的错误路径继续调度。

## Q6：你负责了基于 SSE 的工作流实时状态同步，那从后端视角看，一次工作流执行的状态消息是怎么推到前端的？

**答案：**

整体链路是这样的。

前端发起工作流执行请求后，后端接口会返回一个 `SseEmitter`，也就是建立一条服务端到客户端的单向长连接。接下来工作流不会等全部执行完再一次性返回，而是在执行过程中不断往这条连接里推送事件。

在引擎内部，我把状态通知这件事抽象成了回调接口。工作流开始时会触发 `onWorkflowStart`，节点开始执行时会触发 `onNodeStart`，节点处理中会触发 `onNodeProcess`，结束时会触发 `onNodeEnd`，整个工作流结束时会触发 `onWorkflowEnd`。具体的 SSE 实现类负责把这些事件通过 `SseEmitter` 发给前端。

这样设计的好处是执行逻辑和消息推送逻辑是解耦的。引擎只负责在关键时机抛出事件，不直接关心前端协议细节。对前端来说，它拿到的就是一串有顺序的状态流，可以用来实时刷新节点状态、显示模型流式输出或者展示整体执行进度。

## Q7：LLM 节点和普通插件节点在执行模型上最大的差异是什么？

**答案：**

我觉得最大的差异在于，LLM 节点是“上下文驱动 + 流式输出”的，而插件节点通常是“一次性工具调用 + 结构化返回”的。

LLM 节点执行时，通常要先做模板渲染，把上游变量填进 prompt 里，还可能要拼 system prompt 和历史对话，然后再去调模型服务。调模型的时候还会涉及流式返回，所以节点执行过程中不是只有开始和结束两个状态，中间还会持续往前端推 token 或文本片段。最后还要处理 token usage、思考内容、最终文本输出这些信息。

插件节点相对直接一些，更多是把输入参数交给某个外部工具服务，等它返回结果后写回变量池。它更像 RPC 调用，重点是参数正确性、稳定性和结果映射。

所以如果概括一下，LLM 节点更偏“对话与生成”，插件节点更偏“调用与执行”。

## Q8：如果现在要新增一个“条件判断节点”或者“循环节点”，你觉得当前架构最先会遇到什么挑战？

**答案：**

如果是条件判断节点，挑战还相对可控，主要是要把“节点执行结果如何影响后继路由”这件事设计好。也就是说，执行器不只是返回成功失败，还要返回条件判断结果，然后引擎根据条件选择不同分支。

但如果是循环节点，挑战会明显更大，因为我们当前架构是以 DAG 为前提的，也就是默认图里不能有环。循环节点一引入，首先就会跟现有的环检测规则冲突。其次，节点状态流转、执行次数控制、变量作用域、终止条件这些都要重新设计。

所以如果优先级上来讲，我会先支持条件节点，因为它是在现有 DAG 框架内做分支增强；循环节点则很可能不是加一个执行器就够了，而是要重构调度模型，从“无环图调度”升级到“支持受控回路的流程调度”。

## Q9：这个工作流引擎在工程上最容易出问题的点是什么，你做过哪些保护？

**答案：**

我觉得最容易出问题的地方主要有三类。

第一类是 DSL 本身不合法。比如节点缺失、边引用不存在的节点、工作流里有环，这些如果不提前拦住，执行阶段会非常混乱。所以我们会在执行前做完整性校验和环检测。

第二类是节点执行异常。比如外部模型服务超时、插件调用失败、参数缺失，这些都会影响整个工作流稳定性。针对这类问题，我们通过节点执行状态枚举、失败分支、重试或中断策略来控制流程行为，而不是让异常直接把整个系统打崩。

第三类是状态同步不稳定。比如前端感知不到节点进度、消息顺序乱掉、执行完了但前端还在等。为了解决这个问题，我们把状态推送统一收敛到回调链路里，保证开始、处理中、结束这些关键事件都有明确触发点。

如果再往工程化一点说，我觉得统一 Docker Compose 拉起环境这件事其实也很重要，因为这个项目依赖 MySQL、Redis、MinIO、Console Hub、Workflow Engine 多个服务，本地环境不统一的话联调成本会很高。

## Q10：如果让你复盘这个项目，你觉得自己做得最好的一点和最想补的一点分别是什么？

**答案：**

我觉得自己做得最好的一点，是把工作流执行和节点扩展这两件事拆清楚了。也就是引擎层负责调度，节点执行器层负责具体业务逻辑，SSE 回调层负责状态同步。这样整个系统职责比较清晰，后续新增节点类型时不会把主流程越改越乱。

我最想补的一点，是当前架构虽然已经能支持基础的 DAG 编排，但在更复杂场景下还有明显提升空间。比如并发节点调度、循环节点支持、LLM 结构化输出、执行链路的可观测性，这些都是后面值得继续做的方向。

如果再给我一段时间，我会优先补可观测性和更复杂节点支持。因为对工作流系统来说，能不能跑起来只是第一步，真正难的是出了问题以后能不能快速定位，以及当业务复杂度上来之后架构还能不能稳住。
