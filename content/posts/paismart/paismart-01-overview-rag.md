## 文件合并过程中，如何保证原子性？如果合并过程中失败了，你有什么回滚机制？

项目中，最关键的合并步骤是调用了 MinIO 的 `composeObject` 接口。这个操作是在 **MinIO 服务端内部**完成的，它本身是一个**原子操作**。这意味着，对于 MinIO 来说，分片合并要么就彻底成功，生成一个完整的最终文件；要么就失败，不产生任何文件。

：**1. 调用 MinIO 合并 -> 2. 清理 MinIO 中的分片 -> 3. 删除 Redis 缓存 -> 4. 更新数据库状态为“已完成”**。

**建立后台校准任务**我会创建一个定时的后台任务（例如每小时执行一次），这个任务专门做“兜底”和“校准”的工作。

它会扫描数据库里所有状态为“合并中”并且“卡住”超过一定时间（比如 1 小时）的记录。
对于每一条记录，它会去 MinIO 查询对应的最终文件是否存在。
如果文件存在，说明当时只是后续步骤失败了。校准任务就会主动完成剩下的工作：将数据库状态更新为“已完成”，并清理分片和 Redis 缓存

## 为什么最终选择了Kafka来实现异步处理？相比于其他消息队列（如RabbitMQ），它有什么优势？

比如说我们在文件上传完成之后，会有很多后续的异步任务，比如提取元数据、生成全文索引、做 AI 向量化之类的处理。这些处理不仅数据量大，而且任务本身也比较消耗资源，属于典型的**流式数据处理场景**

Kafka 本身就非常适合这种高吞吐的数据流。它的架构决定了它可以处理每秒上百万条消息，而且是基于磁盘顺序写入，非常稳定，这跟 **RabbitMQ 那种内存优先、消息读完即删的**机制不太一样。

另外一个很关键的点是 Kafka 的数据持久化能力。它的消息是保存在磁盘上的，而且可以保留很久，比如 7 天。消费者通过 offset 去控制消费进度。这个机制带来了一个特别大的好处：我们可以**随时进行数据回溯**。

比如向量化模型升级了，或者数据处理逻辑修复了 bug，我们只需要新建一个消费组从历史 offset 开始消费，就能批量重跑所有老的文件数据，不用动数据库。

## 在使用Kafka时，你是如何配置来确保消息的可靠传递的

**生产者这块**，我们设置了 acks=all，也就是生产者发送一条消息，必须等到所有同步副本都写成功，才算这条消息真正发出。

**服务端这块**，我们把 topic 的副本数设成了 3，也就是一个 leader + 两个 follower，再加上配置了 `min.insync.replicas=2`，意思是如果只有一个副本存活了（比如只有 leader 没挂），Broker 就会拒绝写入请求。

**消费者这边**，我们没有用 Kafka 默认的自动提交，而是设置了 `enable.auto.commit=false` 转为手动提交。

## 文本分块的策略是什么？你是如何去确定一个最优的分块大小的？

一开始我们实现了**最基础的固定大小分块策略**，直接把整个文本按照设定好的 chunkSize 切段。这种方式简单粗暴，但存在的问题是：容易在句子中间或者段落中间截断，导致语义残缺，对检索的精度会有影响。

后来我们引入了**语义感知分块策略**。这个版本会优先尝试按段落进行切割，如果某个段落太长，我们再细化到句子级别。整体上来说，它更尊重语义结构，能让一个 chunk 更自然地保留上下文，对于检索来说准确性提升非常明显。

目的：emebedding向量化模型能向量化的长度有限，分块能够让语义集中，让检索更加的精准

当然了，这种策略仍然会遇到边界切断的问题。比如一个知识点刚好在两个分块的交界处被拆开了，于是我计划引入**重叠分块机制**。我们让每个分块的前后都有一部分“上下文冗余”，这样， embedding 模型在处理时就不会遗漏关键信息，召回率提高了很多。

___

## 避免OOM问题:父块与子切块

我们自定义了一个 `StreamingContentHandler`，它在每次处理字符的回调函数中，会判断当前内容是否达到了**分块阈值，如果是就马上进行切片、入库**。这样整个过程就不再需要等待解析完成之后一次性处理，而是边读边处理。

为了进一步优化内存使用，我们还增加了一个**实时检测内存使用率的处理**，超过阈值就会触发自动 GC。

## es检索算法设计

我们是先用向量去做初筛，再通过 BM25 的 rescore 对初筛结果**再排序**，这样能把那些**语义匹配 + 关键词命中**的高质量结果排到前面。

除此之外，我们还做了权限控制，比如说用户查询的文档必须是在用户所在的组织标签下，或者必须是公开的资料，或者必须是本人上传的文档等等，这些信息我们会通过 filter 的方式一并传给 Elasticsearch，避免返回用户无权限查看的内容。

KNN:kNN 是 Elasticsearch 的一个**向量相似度搜索功能**。它允许我们搜索‘内容语义’而不仅仅是‘关键词’——比如用问题匹配知识库答案，本质是让搜索引擎具备‘联想’的能力。

BM25:判断文档和搜索关键词的相关性

- 关键词在 当前文档 出现次数越多，得分越高。
- 关键词在 所有文档中越稀有（比如“量子计算机” vs “的”），含金量越高，得分越高。
- 惩罚长文档灌水 —— 比如“区块链”在 10 页的报告中出现 5 次，比在 100 页的教材中出现 5 次更可信。

____

第一阶段，我们会通过向量检索先从知识库中“捞”出一个比较大的候选集，比如是 topK 的 30 倍。这个阶段的目标很明确，就是“求全”。不管搜索的内容标不标准、意思精不精准，只要语义上跟用户的问题相关,我们就都拿出来，不放过任何一个潜在的好结果。

第二阶段，我们用调用 Elasticsearch 的 rescore 机制。也就是说，我们不再对所有文档做关键词匹配，而只是对刚刚召回的那一批候选集，再做一次 BM25 的关键词打分。这个阶段的目标是“求准”了—— 那些跟用户关键词完全匹配的结果，或者说那些“说到点子上”的结果，会被调高排名。

**召回加重排序**

## 那它和我们更常用的HTTP请求相比，有什么本质区别？为什么在这个场景下，必须用WebSocket？

**WebSocket 能够支持服务端主动、实时地向客户端推送数据**，而 HTTP 不具备这个能力。

具体来说，HTTP 是一种无状态、单向的请求-响应协议，它的工作机制决定了只能由客户端发起请求，服务端只是被动响应。哪怕我们用长轮询等手段模拟实时性，本质上还是客户端不断地问“有没有新的消息”，服务器无法主动发“有了”。

**全双工的协议，一旦连接建立，前后端就可以随时互相发送消息。**

___

这个 RAG 项目如果升级成 Agent，不是只加一个模型就行，核心是补齐**决策和执行闭环**。  
现在 RAG 是‘检索一次+回答一次’，Agent 要变成‘理解任务→规划步骤→调用工具→观察结果→继续执行→最终交付’。  
所以我会重点补三层能力：

1. **编排层**：加状态机和 Planner，让系统支持多步任务，而不是单轮问答。
2. **工具层**：把知识库检索、文档读取、外部 API 等能力工具化，并做参数校验、权限控制。
3. **保障层**：加可观测日志、失败重试、注入防护和人工确认，保证可控、可审计。

落地顺序我会先做最小闭环：单 Agent + 2 个工具 + 状态流可视化，跑通后再扩展多工具和复杂任务。  
这样改造风险小、上线快，也能量化效果，比如任务成功率和平均步骤数。”

## 后续关于agent的改进

在完成基础 RAG 系统后，我发现传统 RAG 只能被动检索，无法处理需要多步推理的复杂任务，因此将系统升级为 Agent 架构，主要改进包括：

1. 实现**ReAct 推理循环** ：设计最多 4 轮的"思考-行动-观察"循环机制，让 LLM 能根据中间结果动态调整策略，比如"对比产品 A 和 B"会先分别检索再综合对比，而非一次性拼接答案；
2. 构建可扩展的 **Tool Use 框架** ：基于 OpenAI**Function Calling 标准实现工具注册与执行机制**，**目前支持知识库检索、结构化摘要**、用户反馈记录、知识库统计 4 个工具，新增工具只需实现 execute 方法并注册即可；
3. 设计强引导的 Agent Prompt ：通过"默认强制检索 + 白名单豁免"的策略，解决 LLM 自以为知道答案而跳过检索的问题，将检索命中率从 60% 提升至 95% 以上；
4. 实现工具状态实时感知 ：通过 WebSocket 推送工具执行状态（executing/success/failed），对于 generate_summary 等耗时工具实现流式穿透，用户可实时看到"正在检索知识库..."等进度提示；
5. 引入用户反馈闭环 ：新增 submit_feedback 工具记录用户点赞/点踩及原因，在后续对话中作为上下文注入，让 Agent 逐步学习用户偏好，形成持续优化闭环。

## 幂等性

多个用户上传同一个文件：

1. file_upload 是按 fileMd5 + userId 查的，所以不同用户会各自有一条上传记录。redis中的bitmap的key也是会各有一条

2. chunk_info 和 MinIO 路径是**只按 fileMd5 共享**

用户 B 上传用户 A 已上传过的同 MD5 分片，代码会先查 Redis；Redis 是按用户隔离的 key：

但 Redis 未命中后，会查全局 chunk_info：

在 UploadService.java (line 100)。如果 A 已经上传过这个分片，B 会直接认为该分片已上传，然后给 B 的 Redis bitmap 回填并返回，不再上传 MinIO。

这就是它的“幂等/复用”设计：**同一 MD5 的分片，后来的用户可以复用已有 chunk_info**

- **跨用户同文件上传存在缺陷**：第一个用户合并后删除临时分片，第二个用户可能因为 chunk_info 残留而误判分片已上传，最终合并时找不到 MinIO 临时分片。

改进：合并成功后加入mysql chunkinfo的删除，并且在上传/合并之前先检查是否已经存在filemd5的文件 有则复用

## 数据一致性+幂等性面试话术

梳理了项目里的上传链路：分片上传时是先写 MinIO，再写 MySQL 的 chunk_info，最后写 Redis bitmap。也就是说 Redis 只是上传进度缓存，MySQL 才是判断分片是否可合并的事实源。

一开始我发现一个问题：如果 Redis 丢了，但 MySQL 和 MinIO 里其实已经有分片，系统不能简单认为分片没上传。于是我做了兜底逻辑：Redis 未命中时回查 MySQL 的 chunk_info，如果 MySQL 有记录，就回填 Redis 并按幂等成功处理。这样解决了 Redis 缓存丢失导致重复上传或进度异常的问题。也会有这种情况：只有minio中有记录，这种情况的话就会直接认为该分片没有上传可以重传。

___

但继续看代码后，我又发现另一个更隐蔽的问题：不同用户上传同一个文件时，file_upload 是按用户隔离的，但 chunk_info 和 MinIO 路径是按 fileMd5 共享的。第一个用户合并成功后，系统会删除 MinIO 里的临时分片，但没有删除 MySQL 的 chunk_info。这样第二个用户上传同一个文件时，系统查到 chunk_info 存在，**会误以为分片已上传**；但实际 MinIO 临时分片已经被删了，最后合并时就会失败。

修复1：在文件合并成功、确认 merged/{fileMd5} 已经生成之后，同步清理 MySQL 里的 chunk_info 分片记录，避免它继续作为错误的事实源。这样就保证 MinIO 临时分片和 MySQL 分片元数据的生命周期一致。；

最后这个修复把逻辑收敛成：Redis 可以丢，因为能从 MySQL 回源；但 MySQL 的分片记录必须准确反映 MinIO 临时分片是否存在。**合并完成后，临时分片被清理，对应的 chunk_info 也必须清理**，避免跨用户同文件上传时出现错误幂等。

我的最终方案是：**合并成功后，先确认 merged/{fileMd5} 已经生成；然后删除 MinIO 临时分片，删除 MySQL 的 chunk_info，删除 Redis bitmap**最后把当前用户的 file_upload.status 更新为 COMPLETED。这样临时分片、分片元数据、上传进度缓存三者的生命周期就一致了。**但是没有使用秒传和复用提升效率**

同时我又补了一个上传/合并入口的复用判断：如果 merged/{fileMd5} 已经存在，说明这个内容相同的文件已经被其他用户或之前的请求合并过了，那当前用户就不需要再依赖 chunk_info 或重新合并分片，而是直接复用 merged/{fileMd5}，创建或更新当前用户自己的 file_upload 记录为 COMPLETED。

这样修完之后，逻辑就更闭环了：上传过程中 Redis 可以作为缓存丢失，因为能从 MySQL 回源；合并完成后，chunk_info 不再残留，避免误判临时分片还存在；而同 MD5 文件再次上传时，可以直接复用已合并文件，实现更合理的幂等和秒传效果。

## 抗高并发

#### 1.做限流（重点)

按接口成本和用户维度做分层限流

 **按位置分层限流**

> **第一层**可以放在网关或 Nginx，比如限制单 IP 的请求频率，防止恶意刷接口。
> 
> 做一个网关层做ip层或者userid层面的限流  
> **第二层**放在后端 Spring Boot 里，按用户 ID、组织 ID、接口类型做业务限流。
> 
> 接口层面的限流 比如说：/chat接口 按userid+api做用户级别的限流 比如每个用户每分钟最多问10次
> 
> **第三层**放在下游资源前，比如大模型调用、Embedding、文件解析 Kafka 消费者，也要限制并发和速率。
> 
> 大模型调用加**并发限制**，比如单个用户最多同时 2 个生成任务

常见的限流策略：

**固定窗口**：实现简单，比如每分钟最多 60 次，但窗口边界可能突刺
**滑动窗口**：更平滑，适合用户级接口限流
**令牌桶**：允许一定突发流量，常用于普通 API
**漏桶**：稳定匀速处理，适合削峰
**并发数限制**：限制同一时间正在处理的请求数，适合 AI 问答、文件解析
